# AHVS v2.1 — 解释性异常检测器

> Anti-Hallucination Verification System with Temporal Causal Reasoning + Perspective Shift
>
> v1.0: 静态事实核查（对不对）
> v2.0: 时间因果推理（为什么 + 什么会让它失效 + 盯住什么）
> v2.1: 换位思考（站在不同角色视角，发现同一框架内看不到的盲区）
>
> 核心理念：不预测未来，而是找到能让你当前判断失效的东西
>
> 设计依据：两次五方 AI 辩论共识（Opus+Sonnet+Gemini+Codex+Kimi, 2026-02-25）

## 调用方式

```
/ahvs {声明}              # 自动判断：硬事实→v1.0，趋势/时效→v2.0
/ahvs:verify {声明}       # 强制 v1.0 静态核查
/ahvs:temporal {声明}     # 强制 v2.0 时间因果分析
/ahvs:quick {声明}        # 快速版（30秒，1-2次搜索）
/ahvs:deep {声明}         # 深度版（3-5分钟，多源+交叉验证）
```

## v1.0 静态核查（保持不变）

### Step 1: 解析待验证声明

分类为：
- **硬事实**：数字、日期、API、版本、价格（必须验证）
- **软声明**：观点、趋势、推测（标注为观点）

### Step 2: 多源验证

```javascript
// 1. 官方源验证（最高优先级）
WebFetch({ url: "官方文档/API文档" })

// 2. 搜索引擎验证
WebSearch({ query: "声明内容 + 2025/2026" })

// 3. 代码验证（如果涉及技术）
Bash("检查本地文件/配置")
```

### Step 3: 输出验证报告

| 声明 | 状态 | 证据 | 来源 |
|------|------|------|------|
| xxx | ✅ 已验证 / ⚠️ 部分正确 / ❌ 错误 / 🔍 无法验证 | 具体证据 | URL |

### 验证标准

| 状态 | 条件 |
|------|------|
| ✅ 已验证 | 2个以上独立来源确认 |
| ⚠️ 部分正确 | 核心正确但细节有误 |
| ❌ 错误 | 与官方/权威来源矛盾 |
| 🔍 无法验证 | 缺乏可靠来源 |

## v2.0 时间因果分析（新增）

### 领域自动识别

| 关键词 | 领域 | 额外数据源 |
|--------|------|-----------|
| 趋势/风格/面料/色彩/秀场 | fashion_trend | trend-scout 本地数据 |
| 平台/政策/规则/封号/限流 | platform_policy | 官方公告历史 |
| 价格/定价/成本/利润/均价 | market_pricing | competitor-scout 数据 |
| 供应链/面料/交期/供应商 | supply_chain | 采购记录 |
| API/版本/框架/工具 | technology | 官方文档 + GitHub |
| 其他 | general | WebSearch |

### 执行协议：三步法

#### Step 1: NOW — 当前判断 + 证据基线

**目标**：确认当前事实状态，收集历史证据建立基线。

**执行**：

```python
# 1. 解析声明，拆为可验证子声明
# 原始声明: "quiet luxury 已经过时了"
# 子声明: ["quiet luxury 当前搜索趋势", "秀场出现频率", "社媒讨论量变化"]

# 2. 验证当前状态（复用 v1.0 逻辑）
WebSearch(query="{主题} 2026 当前 现状")
WebFetch(url="相关官方/权威来源")

# 3. 回溯历史基线（新增）
mcp__memory-anchor__search_rules(
    query="{主题} 历史 趋势",
    start_time="2023-01-01T00:00:00",  # 回溯 2-3 年
    end_time="2026-02-01T00:00:00"
)
WebSearch(query="{主题} 2023 2024 2025 趋势变化")

# 4. 补充：检查本地数据（按领域）
# fashion_trend → ~/ai_fashion_data/trends/
# market_pricing → ~/ai_fashion_data/competitor/
```

**证据质量标准**：

| 来源类型 | 权重 | 时间衰减 |
|---------|------|---------|
| 官方数据/报告 | 最高 | 1年内全权重，1-3年半权重 |
| Memory Anchor 记录 | 高 | 6个月内全权重，之后衰减 |
| 权威媒体 (BoF/Vogue/WWD) | 中高 | 1年内全权重 |
| 一般网页 | 中 | 6个月内有效 |
| AI 内部知识 | **零** | **禁止当证据用** |

**输出**：

```markdown
### NOW — 当前判断

**声明**: {原始声明}
**当前状态**: {verified/partial/refuted/uncertain}
**判断**: {一句话当前结论}

**证据基线** ({N} 条)：
| # | 事实 | 时间 | 来源 | 权重 |
|---|------|------|------|------|
| 1 | {fact} | {date} | {url} | {高/中/低} |

**惯性评估**: {当前趋势的延续性强不强？基于历史数据判断}
```

**硬规则**：证据不足 3 条时，标注 `⚠️ 证据不足，以下分析基于有限信息`。

---

#### Step 2: PRE-MORTEM — 什么会让判断失效？（含换位思考）

**目标**：反转思考——假设当前判断是错的，找出最可能的原因。同时覆盖正向机会。**v2.1 新增**：通过 JTBD 四力换位思考，发现同一框架内看不到的盲区。

**核心问题**（必须逐一回答）：

```
❶ 假设当前判断是错的，最可能的 3 个原因是什么？（负向偏差/风险）
❷ 假设趋势突然加速，最可能的原因是什么？（正向偏差/机会）
❸ 历史上有没有类似的翻转案例？（历史先例）
❹ 【换位思考】站在不同角色视角，他们怎么看这个判断？（v2.1 新增）
```

**执行**：

```python
# 1. 搜索反面证据
WebSearch(query="{主题} 问题 风险 失败 变化")
WebSearch(query="{主题} 相反观点 批评")

# 2. 搜索历史反转先例
mcp__memory-anchor__search_rules(
    query="{主题} 失败 反转 变化",
    include_expired=True  # 包含过期记忆，找历史教训
)

# 3. 正向机会搜索
WebSearch(query="{主题} 加速 爆发 机会 2026")

# 4. 【v2.1】换位思考：加载角色数据
mcp__memory-anchor__search_rules(
    query="{主题} 客户 用户 角色 persona",
    limit=5
)
# 基于 Memory Anchor 中的真实数据构建角色，不凭空编造
```

##### 换位思考协议（v2.1 — JTBD 四力圆桌）

**认知科学依据**：Galinsky（哥伦比亚商学院）研究证实 perspective-taking 显著降低锚定偏差。但 LLM 角色扮演准确率上限 69%（PersonaEval 2025），因此必须用证据锚定角色，不能凭空想象。

**方法论**：借鉴 JTBD Four Forces 框架，强制生成 4 个互斥视角（MECE），避免"多个角色说同一件事"。

| 力量 | 角色定位 | 核心台词模板 | 示例 |
|------|---------|-------------|------|
| **Push（推力）** | 现状有多糟 — 代表**现有客户/用户** | "我用现在的方案，最痛苦的是…" | 小红书消费者："搜到的都是同质化内容" |
| **Pull（拉力）** | 新方案有多诱人 — 代表**潜在新客户群** | "如果有人能…，我立刻转过去" | 小电商卖家："如果有便宜的AI生图，我不用找摄影师了" |
| **Anxiety（焦虑）** | 会出什么错 — 代表**监管者/保守派/风险方** | "这个方案最可能在…上翻车" | 平台审核："AI生成内容不标注会被封号" |
| **Habit（惯性）** | 为什么不改 — 代表**竞品客户/沉默多数** | "我知道有更好的，但我懒得换因为…" | 竞品用户："虽然贵但我已经习惯了这个供应商" |

**执行规则**：

- **Quick 模式**：只用 Push + Anxiety 两个角色（最关键的一正一反），单次 Council Prompt 生成
- **Deep 模式**：四个角色全用，每个角色必须锚定至少 1 条 Memory Anchor 或 WebSearch 证据
- **硬上限**：最多 4 个角色，超过会降低决策质量（认知科学共识：>5 视角 → 分析瘫痪）
- **每个角色必须回答**：
  1. 站在我的立场，当前判断对我意味着什么？
  2. 我有什么信息是分析者看不到的？
  3. 什么情况下我会离开/转向/反对？
- **Council Prompt（单次生成，不串行调用）**：

```
模拟一场圆桌对质。与会者是 [{Push角色}]、[{Pull角色}]、[{Anxiety角色}]、[{Habit角色}]。
针对判断"{当前判断}"，每人说一句最尖锐的话（不要客套，直接冲突）。
然后输出：他们之间最大的分歧点是什么？这个分歧是否在 Step 2 的风险/机会中已被覆盖？
如果未被覆盖 → 这就是换位思考发现的新盲区，必须补充到风险或机会列表中。
```

- **反身悖论防护**（Sonnet 核查发现）：LLM 角色扮演本身会引入偏见。因此每个角色的"台词"后面必须标注 `[锚定: {数据来源}]` 或 `[无锚定: 纯推理]`。无锚定的台词权重降半。

**输出**：

```markdown
### PRE-MORTEM — 威胁与机会清单

#### 风险（如果判断失效…）

| # | 威胁 | 触发条件 | 信心 | 证据 |
|---|------|---------|------|------|
| 1 | {threat} | 如果{condition}发生 | [有先例] | {source} |
| 2 | {threat} | 如果{condition}发生 | [有迹象] | {source} |
| 3 | {threat} | 如果{condition}发生 | [纯推测] | 无直接证据 |

#### 机会（如果趋势加速…）

| # | 机会 | 触发条件 | 信心 | 证据 |
|---|------|---------|------|------|
| 1 | {opportunity} | 如果{condition}发生 | {label} | {source} |

#### 换位思考圆桌（v2.1）

| 角色(JTBD力量) | 一句话 | 锚定 | 是否新盲区 |
|---------------|--------|------|-----------|
| {Push: 现有客户} | "..." | [锚定: MA记录/WebSearch] | 是/否 |
| {Pull: 潜在新客群} | "..." | [锚定: ...] 或 [无锚定] | 是/否 |
| {Anxiety: 风险方} | "..." | [锚定: ...] | 是/否 |
| {Habit: 惯性方} | "..." | [锚定: ...] | 是/否 |

**圆桌最大分歧**: {一句话描述}
**新发现的盲区**: {补充到风险或机会列表，或"无新盲区"}

#### 历史先例

| 类似案例 | 时间 | 结果 | 与当前的相似度 |
|---------|------|------|---------------|
| {case} | {year} | {outcome} | {高/中/低} |
```

**信心标签定义（替代数字概率）**：

| 标签 | 含义 | 对应证据要求 |
|------|------|-------------|
| **[有先例]** | 历史上发生过类似情况 | 至少1个具体历史案例 |
| **[有迹象]** | 当前有微弱信号但尚不确定 | 至少1条当前数据支撑 |
| **[纯推测]** | 逻辑上可能但无直接证据 | 标注"基于类比推理" |
| **[已验证]** | 正在发生（用于现在时态） | 2条以上独立来源 |

**禁止**：给出数字概率（如"70%可能"）。LLM 概率未经校准，会产生虚假精度。

---

#### Step 3: WATCH LIST — 监控什么信号？

**目标**：从 Step 2 的威胁和机会中，提取可观测的具体指标。

**输出**：

```markdown
### WATCH LIST — 持续监控

| 信号 | 观测方式 | 如果触发 | 复查频率 |
|------|---------|---------|---------|
| {observable_signal} | {how_to_check} | 重新评估{judgment} | {weekly/monthly} |

**建议复查日期**: {date}
```

**硬规则**：每个信号必须是**可观测的**（能用 WebSearch 或数据查到的），不能是"如果市场变了"这种模糊表述。

### 快速版 vs 深度版

| 维度 | /ahvs:quick | /ahvs:deep |
|------|-------------|------------|
| 搜索次数 | 1-2 次 | 5-10 次 |
| 历史回溯 | 仅 Memory Anchor | Memory Anchor + 多年份 WebSearch |
| Pre-mortem | 1-2 个威胁 | 3+ 威胁 + 机会 + 历史先例 |
| 换位思考 | 2 角色（Push+Anxiety） | 4 角色（JTBD 四力全开） |
| Watch List | 1-2 个信号 | 3-5 个信号 |
| 耗时 | 15-40 秒 | 2-5 分钟 |
| Token | ~12k-25k | ~50k-100k |
| 适用场景 | 日常快速决策 | 重大投入/战略方向 |

**自动升级触发**（quick → deep）：
- 涉及金额 > ¥10,000
- 涉及法律/政策/合规
- Step 1 搜索结果互相矛盾
- 证据不足 3 条

## 存储协议（Memory Anchor）

每次 AHVS 分析完成后，按以下规则存入 Memory Anchor：

### 1. 报告摘要
```python
mcp__memory-anchor__add_memory(
    content="AHVS分析: {声明} → {判断状态}。关键发现: {一句话}。换位思考盲区: {有/无, 简述}",
    category="verified_fact",
    confidence=0.9,
    # tags 通过 content 语义检索，无需显式 tag
)
```

### 2. Watch List 项（带过期时间）
```python
mcp__memory-anchor__add_memory(
    content="AHVS监控: {信号} | 触发行动: {action} | 复查: {date}",
    category="verified_fact",
    confidence=0.8,
    # Memory Anchor 内置去重（cosine > 0.92 拦截）
)
```

### 3. 换位思考角色数据（可复用）
```python
# 首次分析某角色时存入，后续复用
mcp__memory-anchor__add_memory(
    content="AHVS角色: {角色名}({JTBD力量})。画像: {叙事描述}。已验证痛点: {list}。历史推演: {上次分析发现了什么}",
    category="customer",  # 或 "competitor" / "regulation"
    confidence=0.8,
)
```

### 4. 下次调用时自动加载
```python
# Step 2 执行时自动搜索已有角色
mcp__memory-anchor__search_rules(
    query="AHVS角色 {主题相关关键词}",
    limit=5
)
# 如果有匹配 → 加载历史角色画像，避免每次重建
# 如果无匹配 → 基于当前上下文构建临时角色
```

**去重保护**：Memory Anchor 内置 cosine similarity > 0.92 去重，重复分析自动拦截。

## 红线规则

### 从 v1.0 保留
- **禁止**：未经验证就说"已验证"
- **禁止**：用 AI 知识代替实际搜索
- **必须**：提供可点击的来源链接
- **必须**：区分"事实"和"观点"

### v2.0 新增
- **禁止**：给出数字概率（如"70%可能"）— LLM 概率未经校准
- **禁止**：没有证据就编场景（无证据 = 标注[纯推测]）
- **禁止**：跳过 Step 1 直接推演未来（没有基线的推演是幻觉）
- **必须**：每条推理锚定至少 1 个外部来源
- **必须**：呈现反面证据（不能只列支持的）
- **必须**：末尾声明"AI辅助推理，非统计预测"
- **必须**：证据不足时诚实说"证据不足"

### v2.1 换位思考新增
- **禁止**：凭空编造角色画像（每个角色必须锚定 Memory Anchor 或 WebSearch 数据）
- **禁止**：超过 4 个角色（认知科学：>5 视角降低决策质量）
- **禁止**：串行生成角色分析（用 Council Prompt 单次生成，控制延迟）
- **必须**：每个角色台词标注 `[锚定: 来源]` 或 `[无锚定: 纯推理]`
- **必须**：无锚定台词权重降半（LLM 角色扮演准确率仅 69%，PersonaEval 2025）
- **必须**：检查换位思考是否发现了前三步没有的"新盲区"，如果没有则明确标注"无新盲区"
- **必须**：Quick 模式只用 Push + Anxiety 两个角色，不超过 500 token 增量

### 示例 1: v1.0 静态核查

```
用户: /ahvs 小红书2026年禁止AI生成内容

→ v1.0 路由（硬事实声明）
→ WebSearch + WebFetch
→ 结果: ⚠️ 部分正确 — 要求标注但未全面禁止
```
